## CSR (Client-Side Rendering, 클라이언트 사이드 렌더링)

CSR은 대부분의 렌더링을 브라우저(클라이언트)에서 처리합니다.

브라우저가 서버에 페이지를 요청합니다.

서버는 거의 비어있는 HTML 파일과 웹사이트를 동작시키는 **JavaScript 파일(번들)**을 보냅니다.

브라우저가 JavaScript 파일을 다운로드하고 실행합니다.

JavaScript(예: React, Vue)가 API를 통해 필요한 데이터를 요청하고, 브라우저에서 HTML을 동적으로 생성하여 사용자에게 보여줍니다.

장점:

빠른 페이지 전환: 초기 로딩 후, 다른 페이지로 이동할 때 필요한 부분만 다시 그리기 때문에 네이티브 앱처럼 빠르고 부드럽게 느껴집니다. (서버에 매번 HTML을 요청할 필요 없음)

서버 부담 감소: 서버는 정적 파일과 데이터(JSON)만 제공하면 되므로 부담이 적습니다.

단점:

느린 초기 로딩 속도 (TTI): 사용자가 첫 화면을 보기까지 JavaScript 번들을 다운로드하고 실행하는 시간이 필요해 초기 로딩이 느릴 수 있습니다.

SEO 문제: 검색 엔진 로봇이 비어있는 HTML을 보게 될 수 있어, 검색 엔진 최적화(SEO)에 불리할 수 있습니다. (최근 구글 봇은 개선되었지만 여전히 이슈가 될 수 있음)

## SSR (Server-Side Rendering, 서버 사이드 렌더링)

SSR은 페이지 렌더링을 서버에서 처리합니다.

브라우저가 서버에 페이지를 요청합니다.

서버에서 필요한 데이터를 모두 가져와 완성된 HTML 페이지를 만듭니다.

서버가 이 완성된 HTML을 브라우저로 보냅니다.

브라우저는 즉시 내용을 표시합니다. (빠른 첫 화면 로딩, FCP)

(이후) 브라우저가 JavaScript 파일을 다운로드하여 HTML에 이벤트 핸들러 등을 연결해 페이지를 상호작용 가능하게 만듭니다. (이 과정을 **"Hydration"**이라고 함)

장점:

빠른 초기 로딩 속도 (FCP): 사용자가 완성된 페이지를 바로 볼 수 있어 초기 로딩이 매우 빠릅니다.

SEO 친화적: 검색 엔진 로봇이 처음부터 모든 콘텐츠가 채워진 HTML을 받으므로 SEO에 유리합니다.

단점:

느린 페이지 전환: 다른 페이지로 이동할 때마다 서버에 새로운 HTML을 요청하고 받아와야 해서 CSR보다 깜빡임이 발생하거나 느리게 느껴질 수 있습니다.

서버 부담 증가: 모든 사용자의 요청에 대해 서버가 매번 HTML을 생성해야 하므로 서버 부하가 커집니다.
